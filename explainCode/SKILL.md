# Explain Code

当用户说“解释这个 bug”“讲讲这个 bug 怎么回事”“我没看懂你刚才的改动”“带我理解这个修复”或表达相同意图时，使用这个 skill。

目标不是简短总结，而是把用户当成刚接手代码的新手开发，让他最终能真正理解：

1. 这个流程本来是怎么设计的
2. 代码里这个流程是如何落地的
3. bug 是在哪个环节偏离了设计
4. 我们具体改了什么
5. 为什么这样改能修好问题

## 适用场景

在以下情况优先使用：

- 用户要求“解释这个 bug”或同义表达
- 用户说“我没看懂你刚才的修复”
- 用户希望从流程角度理解一次改动
- 用户是新手，需要从大概念一路讲到代码细节
- 用户想知道“为什么会错”以及“为什么这样修”

## 核心要求

解释时必须同时满足下面几点：

1. **以设计者视角讲**
   先讲这个模块或流程为什么要这么设计，它解决什么问题，再讲实现细节。

2. **默认用户是开发新手**
   不要偷跳步骤，不要直接扔术语。必须先建立概念，再进入实现。

3. **从大到小展开**
   固定按下面顺序组织：
   - 目标与职责
   - 整体流程
   - 关键代码入口
   - 关键分支和状态变化
   - bug 触发点
   - 修复点
   - 修复后的正确流程

4. **每一句解释都要能在代码中找到依据**
   任何关键结论都要落到具体文件、函数、条件分支、变量、状态或调用链上。
   不要只讲抽象理解，不要只讲“应该是这样”。

5. **解释的目的是让用户看懂刚才的改动**
   最终要把“旧逻辑”和“新逻辑”对照讲清楚，让用户知道改动为什么必要。

## 解释前必须做的事

在开始回答前，先基于当前仓库和本轮上下文确认这些信息：

1. 这次 bug 涉及的入口函数或入口文件是什么
2. 关键调用链是什么
3. bug 发生前的旧逻辑是什么
4. 本次改动具体改了哪些代码
5. 修复后行为和修复前相比有什么差异

如果信息不足，不要编造。先补充阅读相关代码，再解释。

## 解释方法

### 第一步：先讲设计目标

先用新手能理解的话回答：

- 这个流程是为了解决什么问题
- 流程里每个关键角色分别负责什么
- 数据、状态或事件是怎么流动的

这一部分不要一上来就贴代码，但必须能自然过渡到代码入口。

示例句式：

- “这个流程的设计目标是……，所以它被拆成了……几个阶段。”
- “从职责上看，`A` 负责接收输入，`B` 负责做判断，`C` 负责真正落地执行。”
- “也就是说，正常情况下，请求会沿着 `入口 -> 校验 -> 决策 -> 执行` 这条路径往下走。”

### 第二步：把抽象流程映射到代码

把上面的设计目标映射到真实代码：

- 入口在哪个文件、哪个函数
- 流程的下一跳是谁调用谁
- 哪些条件分支决定了后续行为
- 哪些状态、字段、参数是关键证据

要求使用代码锚点，例如：

- `src/foo.ts` 里的 `handleSubmit()` 是入口
- `service/bar.js` 里的 `buildPayload()` 决定了关键参数
- `if (xxx)` 这个分支控制是否进入异常路径

如果合适，可以用一段简短链路表示：

`A() -> B() -> C() -> D()`

### 第三步：解释 bug 是怎么形成的

这里不要直接说“这里写错了”。要按因果链解释：

1. 原设计希望发生什么
2. 旧代码实际上做了什么
3. 哪个条件、状态或调用顺序偏离了设计
4. 这个偏离是如何一步步放大成最终 bug 的

必须指出真正的触发点，而不是只描述最终现象。

优先使用这类表达：

- “设计预期是……，但代码在 `xxx` 这里实际判断成了……。”
- “问题不是最后报错这一行本身，而是更早的 `xxx()` 已经把状态带偏了。”
- “当 `yyy` 条件成立时，流程进入了本不该进入的分支，所以后面 `zzz` 的前提就失效了。”

### 第四步：解释我们是怎么修的

修复说明必须包含：

1. 改动点在哪里
2. 改动前后的区别是什么
3. 这个改动纠正的是哪一个错误前提
4. 为什么它能恢复原本的设计意图

不要只说“加了判断”“改了顺序”“修了参数”。
要说明：

- 为什么这个判断原来缺失会出错
- 为什么顺序错了会影响结果
- 为什么参数值不对会把流程导向错误路径

### 第五步：讲修复后的完整流程

最后要重新走一遍流程，但这次走的是“修复后的正确路径”。

让用户能看见：

- 输入进来后先到哪里
- 哪个条件现在被正确处理
- 哪个状态现在被正确传递
- 最终为什么结果恢复正常

## 输出结构

默认按这个结构回答：

### 1. 一句话先说结论

用一句话概括：

- bug 的本质是什么
- 修复的本质是什么

### 2. 这个流程本来是怎么设计的

从职责和阶段角度解释，不急着钻进细节。

### 3. 代码里这个流程是怎么走的

按调用链或执行顺序展开，并引用关键文件和函数。

### 4. bug 是在哪里产生的

指出具体偏离点，并解释它如何传播成最终问题。

### 5. 我们改了什么

对照讲清楚旧逻辑和新逻辑。

### 6. 为什么这样改就修好了

把修复点和设计目标重新接起来。

### 7. 你可以重点记住什么

给新手一个可复用的理解框架，例如：

- 以后遇到同类问题应该先看哪个入口
- 应该先核对哪个状态或条件
- 应该怎样判断是“现象”还是“根因”

## 表达规范

- 优先使用短句，少堆术语
- 必须解释术语，不要默认用户知道
- 可以类比，但类比后必须回到代码
- 不要只贴大段代码，要解释代码在流程中的职责
- 不要泛泛而谈，每一段都要能对应到实际代码位置
- 如果本轮对话里已经做过修改，要明确区分“改之前”和“改之后”

## 禁止事项

- 不要只给结论，不讲推导过程
- 不要只讲现象，不讲根因
- 不要只讲代码行数，不讲流程意义
- 不要把用户当成熟悉上下文的老开发
- 不要编造设计意图；如果设计意图不明确，就基于代码和行为做保守说明
- 不要省略“为什么这样修有效”这一段

## 回答模板

可直接按下面模板组织回答：

```md
一句话结论：这个 bug 的本质是 ……；这次修复的本质是 ……。

1. 这个流程本来是怎么设计的
先解释这个模块要解决什么问题，再说明正常流程应该怎么走。

2. 代码里它具体怎么跑
- 入口：`fileA:fnA`
- 下一步：`fileB:fnB`
- 关键判断：`fileC` 里的 `if (...)`
- 关键状态/参数：`xxx`

3. bug 是怎么发生的
设计预期是 ……，但旧代码在 `fileD:fnD` 里做成了 ……。
这会导致流程从 …… 偏到 ……，最终表现成 ……。

4. 我们改了什么
改动点在 `fileE:fnE`。
改之前：……
改之后：……

5. 为什么这样改能修好
因为它把原来错误的 …… 修正成了 ……，所以后续流程重新回到设计预期。

6. 修复后的流程再走一遍
现在输入进入 `...` 后，会先 ……，然后 ……，最后 ……。

7. 你可以怎么记
以后看这类 bug，先看入口、再看关键分支、最后看状态是否沿链路被正确传递。
```

## 成功标准

这个 skill 生效时，回答应满足以下标准：

1. 用户即使不熟悉这段代码，也能跟着理解流程
2. 回答中每个关键结论都能回指到代码证据
3. 用户能区分“设计目标”“实现路径”“bug 成因”“修复动作”四件事
4. 用户能理解为什么这次改动是必要且正确的
