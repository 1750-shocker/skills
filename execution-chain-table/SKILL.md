# Execution Chain Table

当用户想建立 bug 所在执行链认知，或要求“把流程节点串起来”“旁边写代码链”“用表格梳理加载逻辑/过滤逻辑/旧代码链路/新代码链路”时，使用这个 skill。

核心输出是两列并排表：

| 流程节点 | 代码链 |
|---|---|
| 业务上发生了什么 | 具体文件:行号、方法、字段、条件分支如何串起来 |

## 文件保存约定

当用户表达“梳理流程到表格”“整理成表格”“写进 md”“保存表格”或类似意思时，除了在回复中给出核心结论，还要把表格保存为 Markdown 文件。

- 用户指定保存路径：严格写入用户指定路径。
- 用户未指定保存路径：默认保存到当前会话场景对应 Jira 票的 bug 目录。
- Jira 票号来源优先级：用户当前消息中的票号 > 本轮上下文最近明确票号 > 当前分析目录名。
- 默认目录格式：`D:/logAnalyseForRag/<Jira票号>/`，例如 `D:/logAnalyseForRag/DP-6298/`。
- 默认文件名使用业务主题，格式：`<Jira票号>_<主题>_执行链表格.md`。
- 如果无法确定 Jira 票号，先从当前会话已读文件路径、日志目录、快照目录推断；仍无法确定时再问用户要票号或保存路径。
- 回复结尾必须告诉用户实际写入的绝对路径。

## 适用场景

优先用于：

- 用户对一个 bug 的执行链还没建立认知
- 涉及多个入口、Activity/Fragment/ViewModel/Repository/Service/弹窗/回调
- 需要解释“为什么 A 场景正常，B 场景异常”
- 需要对比旧代码链路和修复后链路
- 需要把日志现象映射到代码执行路径
- 用户明确喜欢或要求“表格”“纯文字流程 + 代码链”

不适合：

- 只有一个函数内的简单 bug
- 用户只要最终 commit message
- 用户明确要求代码实现，不需要解释链路

## 输出原则

1. **左列写人能理解的流程节点**
   - 不写代码细节
   - 按业务时序描述：进入页面、点击按钮、请求接口、过滤列表、保存状态、弹窗加载更多
   - 每行只表达一个因果节点

2. **右列写可验证的代码链**
   - 必须包含文件名和行号
   - 写清方法名、字段、关键判断、关键赋值
   - 用 `->` 串起调用或状态流转
   - 不要只写“某方法处理了”，要写具体代码动作

3. **按链路分组，不把所有节点塞进一张表**
   - 每个入口或子系统一张表
   - 常见分组：详情页链路、弹窗链路、数据容器链路、接口链路、旧代码链路、修复后链路

4. **先讲事实链，再讲结论**
   - 表格先还原流程
   - 表格后再写关键差异、数字链、根因结论
   - 不要先下结论再找证据拼接

5. **不要编造调用链**
   - 没读到代码或日志时，先查代码/日志
   - 推断必须标注“根据命名和上下文推断”
   - 关键断言要能回到文件、行号、日志行或接口参数

## 推荐结构

### 1. 链路 A：主入口流程

用于第一个入口，例如详情页加载、首页点击、播放页打开。

| 流程节点 | 代码链 |
|---|---|
| 用户进入页面，请求第一页数据 | `FooActivity.java:120` `initData()` -> `FooViewModel.kt:80` `requestData(false, id)` |
| ViewModel 判断业务类型，进入列表请求 | `FooViewModel.kt:95` `when(type)` -> `FooViewModel.kt:102` `getPlaylist(id)` |
| Repository 发起接口请求 | `FooRepository.java:66` `getList(id, offset, limit)` |

### 2. 链路 B：异常入口或弹窗流程

用于第二个入口，例如从推荐页直接点歌、弹窗加载更多、后台回调。

| 流程节点 | 代码链 |
|---|---|
| 弹窗打开后读取播放队列 | `PlaylistPopup.java:300` `musicListCallback(list)` |
| 本地列表接收播放队列 | `PlaylistPopup.java:306` `mList.clear()` -> `PlaylistPopup.java:307` `mList.addAll(list)` |
| 用户上拉触发分页 | `PlaylistPopup.java:420` `onLoadMore()` -> `PlaylistPopup.java:423` `loadMore(parentId, type)` |

### 3. 关键差异表

用于解释两个场景为什么行为不同。

| 对比点 | 正常链路 | 异常链路 |
|---|---|---|
| 初始列表来源 | 详情页接口一次拉完整页 | 推荐页只拉首页卡片数据 |
| 分页状态来源 | ViewModel 保存页码 | 弹窗从已有列表反推 |
| 错误触发点 | 无 | offset/状态从错误数量推导 |

### 4. 数字链 / 日志链

当 bug 与数量、页码、offset、过滤数量、时间线有关时，必须单独列数字链。

| 流程节点 | 代码链 |
|---|---|
| 首次请求返回 100 条 raw 数据 | `ViewModel.kt:258` 请求 `offset=0, limit=100`；日志 `raw size=100` |
| 过滤后只剩 98 条 | `ViewModel.kt:268` `filterCannotPlaySong(raw)` |
| 弹窗按旧逻辑请求错误 offset | `Popup.java:1084` `size <= 100` -> `page = 101` |
| 接口返回空或少一条 | 日志 `offset=101, raw size=0/17` |

### 5. 一句话结论

最后用一句话压缩根因：

> 详情页的分页变量来自接口原始页，过滤只影响展示；弹窗拿到的是过滤后的播放队列，所以不能用展示数量直接反推接口 offset。

## 工作步骤

1. **先定位入口**
   - 从用户描述、日志、文件名、现象中找入口
   - 搜索相关 tag、方法名、请求名、字段名

2. **按时序读代码**
   - 入口 -> 分支 -> 数据请求 -> 过滤/转换 -> 状态保存 -> UI/弹窗 -> 后续加载
   - 每一步只读必要范围，不整文件硬扫

3. **提取流程节点**
   - 把代码动作翻译成业务节点
   - 每个节点必须能独立回答“这一步发生了什么”

4. **补齐代码链**
   - 每个流程节点右侧放具体文件、行号、方法和关键表达式
   - 如果链路跨文件，用 `->` 串起来

5. **补一张差异表或数字链**
   - 有多个场景时必须有差异表
   - 有数量/分页/状态变化时必须有数字链

6. **给结论，不提前给方案**
   - 先让用户理解 bug 怎么发生
   - 如果用户追问“怎么改”，再给最小改法

## 质量检查

输出前检查：

- 每一行左列都是流程语言，不是代码摘要
- 每一行右列都有代码锚点或日志锚点
- 表格按真实执行顺序排列
- 正常链路和异常链路没有混在一起
- 数字链中的数字来自日志或代码，不是猜的
- 结论能从表格直接推出

## 常用句式

- “先不用图，用两条并排链路建立认知。”
- “左边是业务流程节点，右边是对应代码链。”
- “这张表只解释执行链，不先讨论修复方案。”
- “关键差异不是最后的异常，而是更早一步的数据来源不同。”
- “这个数字链能闭环：初始数量 -> 接口参数 -> 返回数量 -> 最终现象。”

## 反例

不要这样写：

| 流程节点 | 代码链 |
|---|---|
| 调用 getData | `getData()` |
| 调用 request | `request()` |

问题：左列不是业务流程，右列没有文件、行号、关键状态。

应该这样写：

| 流程节点 | 代码链 |
|---|---|
| 进入歌单详情页，请求第一页歌曲 | `CloudMusicPlaylistDetailActivity.java:311` `getData()` -> `CloudMusicPlaylistDetailActivity.java:314` `requestData(false, type, playlistId, vm)` |
| ViewModel 用接口原始页码计算 offset | `CloudMusicPlaylistDetailViewModel.kt:256` `currentPage++` -> `CloudMusicPlaylistDetailViewModel.kt:257` `offset = currentPage * pageSize` |
