# 自动 Bug 分析工作流

当用户提供 Jira 链接或本地日志目录，并要求自动分析 Bug 时，执行单票分析。当用户表达“开始分析 bug / 自动分析所有 bug / 分析我名下所有 bug”等批量意图时，执行两轮式批量分析。

核心原则：不要只给“能解释现象”的结论，必须持续追到最早可证明的根因；如果追不下去，明确说明缺少什么日志、证据或上下文。

## 路径与工具

- 名下 Jira 清单：`D:\BaiduSyncdisk\笔记\公司备忘\jira tickets.md`
- Bug 根目录：`D:\logAnalyseForRag`
- 单票目录：`D:\logAnalyseForRag\<票号>`
- 单票现象文件：`D:\logAnalyseForRag\<票号>\现象.md`
- 本 skill 脚本目录：`{SKILL_DIR}\scripts`

可用脚本：

```text
{SKILL_DIR}\scripts\fetch_issue_context.py
{SKILL_DIR}\scripts\smart_jira_log.py
{SKILL_DIR}\scripts\range_extract_7z_jira.py
{SKILL_DIR}\scripts\extract_logcat_from_android_zip.py
```

`fetch_issue_context.py` 用于轻量拉取 Jira 标题、描述、评论、Severity 和附件清单。批量模式下日志和附件下载不使用 `smart_jira_log.py` 的最小下载策略，而是全量下载 Jira 附件。

## 总原则

1. 只基于真实日志、代码和用户给出的现象下结论；没有证据的判断只能写入“待确认项”。
2. 分析时必须区分四层：
   - 现象：用户看到什么异常。
   - 直接原因：哪个状态/回调/分支导致异常出现。
   - 上游原因：为什么这个状态会被写错、没清掉、没回调、没互斥。
   - 最早根因：从当前日志和代码能证明的最早异常点。
3. 禁止把“直接原因”包装成“根因”。如果只能解释后半段，必须明确说“目前只解释到直接原因，还需要继续往前追”。
4. 对任何缓存、状态位、全局变量、焦点状态、播放态、登录态、网络态，都必须反查：
   - 最后一次被置为正确值/错误值的日志。
   - 应该被清掉/覆盖/回调但没有发生的日志。
   - 后续代码为什么相信这个缓存。
5. 如果继续分析需要更多日志、源码、时间范围、复现步骤、commit 信息或测试包信息，必须立即告诉用户，不要硬凑结论。

## 批量模式：两轮式自动分析

### 触发条件

用户说以下或同等意图时，进入批量模式：

- “开始分析 bug”
- “自动分析所有 bug”
- “分析我名下所有 bug”
- “把我名下 bug 都分析一下”

### 第一轮：准备与全量下载

第一轮只做准备和下载，不分析日志、不查源码。

1. 先加载并执行 `pull-jira-tickets` skill，刷新 `D:\BaiduSyncdisk\笔记\公司备忘\jira tickets.md`。
2. 从 `jira tickets.md` 的 Markdown task list 中解析所有票号、Jira 链接、状态、标题和原始顺序。
3. 对每张票创建或检查 `D:\logAnalyseForRag\<票号>`：
   - 目录不存在则创建。
   - 目录已存在则保留原内容，不删除、不覆盖用户文件。
   - 检查 `D:\logAnalyseForRag\<票号>\现象.md`，不存在则创建空文件。
4. 对每张票全量下载 Jira 附件到 `D:\logAnalyseForRag\<票号>`：
   - 下载日志、压缩包、图片、视频和其它附件。
   - 不做附件类型过滤。
   - 不使用最小下载策略。
   - 不按故障时间筛选 android 日志。
   - 如果附件中的日志文件或日志压缩包单个文件超过 2GB，则跳过该票，不下载该大日志，标记 `跳过-日志超过2GB`。
5. 第一轮结束后停止分析，并提示用户填写每票 `现象.md`。

第一轮不能进入日志分析、源码分析、根因判断或修复建议。

### 第二轮：读取现象并逐票分析

当用户再次要求继续批量分析或自动分析所有 bug 时：

1. 重新执行 `pull-jira-tickets` 刷新清单，并解析所有票。
2. 对每张票调用 `{SKILL_DIR}\scripts\fetch_issue_context.py <票号> --json` 获取 Severity。
3. 默认排序为 `Severity A -> B -> C -> D -> 未知`，同级保持 `jira tickets.md` 中的原始顺序。
4. 逐票读取 `D:\logAnalyseForRag\<票号>\现象.md`：
   - 文件为空：标记 `跳过-现象.md为空`，继续下一票。
   - 缺少固定字段或必要内容：标记 `跳过-现象信息不完整`，继续下一票。
   - 项目目录不存在、不可访问，或指定模块在项目中找不到：标记 `跳过-项目或模块不存在`，继续下一票。
   - 信息完整：进入单票分析流程。
5. 第二轮遇到单票阻塞时跳过该票，不停止整批。
6. 批量结束后追加总览到 `D:\BaiduSyncdisk\笔记\公司备忘\jira tickets.md`。

### `现象.md` 固定字段

第一轮创建的 `现象.md` 必须是空文件。用户填写时只需要提供以下字段：

```text
项目目录：
模块：
现象描述：
```

字段规则：

- `项目目录` 必填，必须是本机可访问的源码目录。
- `模块` 可为空；为空时默认按“云音乐模块”处理。
- `现象描述` 必填，要求尽量包含准确时间点、操作步骤、用户看到的异常。
- 如果 `现象描述` 没有指定时间点，第二轮不阻塞该票；分析时必须从全量日志中自行寻找符合现象的时间段，并在结论中标明“时间点由日志反查，置信度受影响”。
- `现象.md` 只作为分析入口，不作为根因结论证据；最终结论仍必须由日志和源码支撑。

### 批量总览追加格式

批量结束后，在 `D:\BaiduSyncdisk\笔记\公司备忘\jira tickets.md` 末尾追加新的总览块，不覆盖前面的票据分组，也不删除历史总览。

总览块必须带本次开始时间：

```markdown
# 批量 Bug 分析总览

开始时间：YYYY-MM-DD HH:mm:ss

| 票号 | 状态 | 阻塞原因 | 结论置信度 | 输出路径 |
|------|------|----------|------------|----------|
| BUG-123 | 已分析 | 无 | 高 | D:\logAnalyseForRag\BUG-123 |
```

每票状态只能使用以下之一：

- `已分析`
- `跳过-现象.md为空`
- `跳过-现象信息不完整`
- `跳过-项目或模块不存在`
- `跳过-日志超过2GB`
- `失败-附件下载`
- `失败-日志分析`
- `需要继续追因`

置信度只能使用：`高`、`中`、`低`、`无`。

## 单票分析流程

### 步骤 1：处理输入

1. 判断用户输入是 Jira 链接、Jira 票号，还是本地日志目录路径。
2. 如果既不是 Jira 链接、Jira 票号，也不是本地目录路径，向用户询问一句话：请提供 Jira 链接、Jira 票号或本地日志目录。
3. 如果是 Jira 链接或票号：
   - 提取 Jira 票号。
   - 调用 Jira 工具全量下载附件到 `D:\logAnalyseForRag\<票号>`。
   - 如果下载失败，说明失败原因并停止，不要进入后续分析。
   - 检查 `D:\logAnalyseForRag\<票号>\现象.md`；不存在则创建空文件并提示用户填写。
4. 如果是本地目录：
   - 记录该目录为目标日志目录。
   - 不下载 Jira 附件。
5. 如果目标目录已有历史快照或证据链，先读取作为上下文；不要用 RAG 代替本地文件检查。

### 步骤 2：确认分析输入是否足够

开始分析前，必须确认或记录以下信息：

- Bug 现象。
- 用户观察到的准确时间点；如果没有时间点，则从全量日志中按现象反查。
- 时间误差范围；未提供时默认按 `±1 分钟` 搜索，无法确定时间点时扩大到全量日志。
- 目标日志目录。
- 项目目录。
- 模块；未提供时默认“云音乐模块”。

如果单票模式缺少项目目录、现象描述，或项目目录/模块不可用，必须询问用户补充后再分析。批量模式则标记该票并跳过。

### 步骤 3：检查历史分析

1. 在目标目录查找：
   - `*_context_snapshot.md`
   - `*_evidence_chain.md`
   - `snapshot*.md`
   - 用户指定的历史分析文件
2. 如果存在，必须先读取并总结：
   - 之前的根因判断。
   - 关键日志。
   - 已修改代码。
   - 未确认项。
3. 新分析要明确说明本次与历史分析的相同点、差异点和是否推翻旧结论。

### 步骤 4：日志分析

1. 优先围绕用户时间点按 `±1 分钟` 搜索关键 tag、关键词和业务对象。
2. 如果用户没有提供时间点，必须从全量日志中搜索符合现象描述的事件，并标明时间点由日志反查。
3. 还原事件时间线，至少包含：
   - 用户动作。
   - 关键请求/回调。
   - 状态写入。
   - 状态被覆盖。
   - 异常出现时的当前状态。
4. 对关键状态做反向追踪：
   - 谁第一次写入这个状态？
   - 谁最后一次覆盖它？
   - 本该出现的清理/LOSS/失败/取消/重置日志是否出现？
   - 如果没有出现，继续往更早时间段找。
5. 对并发、竞态、缓存类问题，必须查两个方向：
   - 成功样本中该状态如何正常转换。
   - 失败样本中哪一步没有发生。
6. 如果发现关键事件发生在当前时间窗口之前，必须主动扩大时间范围；不要只分析用户给出的表面异常时间点。

### 步骤 5：源码分析

1. 从日志 tag、类名、方法名反查源码。
2. 优先看和日志直接对应的文件，不做无目的全项目搜索。
3. 每个关键日志必须尽量对应到：
   - 类/方法。
   - 条件分支。
   - 读写的变量或缓存。
   - 对外发送的事件或状态。
4. 如果日志显示状态被信任为真，必须找到代码里“为什么信任它”的判断。
5. 如果代码里有缓存命中、同曲 return、提前 return、焦点本地集合、全局单槽状态等，必须作为重点风险点继续追问来源。

### 步骤 6：根因分层输出

输出结论时必须按以下结构组织：

```text
现象：
直接原因：
上游原因：
最早可证明异常点：
证据：
仍需确认：
```

要求：

- “最早可证明异常点”必须有日志行或源码逻辑支撑。
- 如果只能定位到直接原因，必须写“当前还不是最终根因”。
- 如果需要更早日志，必须写清楚缺少的绝对时间范围和建议搜索关键词。
- 如果需要系统/平台侧确认，必须说明应用日志已经证明了什么、没法证明什么。
- Jira 评论、图片、视频只能作为辅助证据，不能替代日志和源码证据。

### 步骤 7：NG 分析规则

如果用户说明这是 NG 分析：

1. 先分类：
   - A. 原现象完全没变。
   - B. 原现象改善但仍有残留。
   - C. 原现象消失，出现新问题。
2. 必须先验证测试包是否包含修复 commit；无法验证时，把“测试包是否包含 commit”列为最高优先级待确认项。
3. 必须读取原 Bug 快照，并对比：
   - 原关键日志是否仍出现。
   - 原根因链路是否仍成立。
   - 修复代码对应日志是否出现。
   - 新失败点发生在原失败点之前还是之后。

### 步骤 8：保存分析快照

分析完成后加载 `snapshot` skill，并在目标目录保存：

1. `<票号>_context_snapshot.md`
2. `<票号>_evidence_chain.md`

快照必须包含：

- 问题概述。
- 复现条件。
- 时间线。
- 根因分层。
- 关键日志证据。
- 源码链路。
- 修复思路或已实施修复。
- 待确认项。

保存到 `D:\logAnalyseForRag` 或其子目录后，执行知识库更新脚本：

```text
cmd.exe /c "D:\aboutMCP\rag-mcp-server\update_knowledge_base.bat"
```

并把脚本结果告知用户。批量模式下如果脚本失败，写入批量总览。

## 强制追根因规则

每次自动分析必须进行“5 Why”式追问，直到满足以下任一停止条件：

1. 找到最早可证明异常点，并能用日志或源码解释它如何导致用户现象。
2. 已向前追到日志起点，仍缺少关键事件，必须列出缺少的时间段和关键词。
3. 相关模块源码不在当前工作区，必须列出缺少的文件/模块/包名。
4. 系统或外部服务行为无法从应用日志证明，必须明确标为“需要系统/平台侧确认”。
5. 用户明确要求停止继续追因。

输出结论前必须自检以下问题：

- 我解释的是用户现象，还是最早根因？
- 当前结论是否还能继续问“为什么”？
- 如果还能继续问，是否已经查过对应日志和源码？
- 是否存在一个状态被错误缓存、未清理、未互斥、未回调？
- 是否需要更早日志或其他模块日志？如果需要，是否已经明确告诉用户？

## 输出要求

1. 先给结论，再给证据。
2. 结论必须标注置信度：高 / 中 / 低。
3. 关键日志引用要足够精确，包含时间、tag、核心字段。
4. 不要为了快速回答省略“不确定项”。
5. 如果还需要继续追根因，直接说“当前只定位到直接原因，我需要继续查更早日志/某模块源码”，不要假装已经闭环。
6. 批量模式必须在最后给出每票处理状态，并追加写入 `D:\BaiduSyncdisk\笔记\公司备忘\jira tickets.md`。
