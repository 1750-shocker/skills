# 自动 Bug 分析工作流

## 目标

当用户提供 Jira 链接或 Jira 票号，并要求自动分析 Bug 时，执行单票分析。

核心原则：不要只给“能解释现象”的结论，必须持续追到最早可证明的根因；如果追不下去，明确说明缺少什么日志、证据或上下文。

## 输入契约

- 默认只接受 Jira 链接或 Jira 票号。
- 如果用户没有提供 Jira 链接或 Jira 票号，直接询问一句话：请提供 Jira 链接或 Jira 票号。
- 如果 Jira 信息、当前工作目录和日志内容不足以支撑分析流程，向用户寻求帮助，例如询问具体现象和对应详细时间点。

## 路径与工具

- Bug 根目录：`D:\logAnalyseForRag`
- 单票目录：`D:\logAnalyseForRag\<票号>`
- 本 skill 脚本目录：`{SKILL_DIR}\scripts`

可用脚本：

```text
{SKILL_DIR}\scripts\fetch_issue_context.py
{SKILL_DIR}\scripts\smart_jira_log.py
{SKILL_DIR}\scripts\range_extract_7z_jira.py
{SKILL_DIR}\scripts\extract_logcat_from_android_zip.py
```

`fetch_issue_context.py` 用于轻量拉取 Jira 标题、描述、评论、Severity 和附件清单。

除脚本外，必须主动使用当前环境提供的通用能力补齐自动化链路：Jira MCP 工具下载附件、文件/目录读取工具定位附件、归档/压缩包读取或解压工具展开日志、搜索工具筛选关键日志、源码读取/搜索/LSP 工具映射代码。脚本不存在、不可用或能力不足时，不能因此停止；应改用通用工具完成同等目标，并在结果中说明采用的替代方式。

## 执行流程

### 1. 处理输入

1. 判断用户输入是 Jira 链接还是 Jira 票号。
2. 如果既不是 Jira 链接也不是 Jira 票号，向用户询问一句话：请提供 Jira 链接或 Jira 票号。
3. 如果是 Jira 链接或票号：
   - 提取 Jira 票号。
   - 调用 Jira 工具全量下载附件到 `D:\logAnalyseForRag\<票号>`。
   - 如果下载失败，说明失败原因并停止，不要进入后续分析。
4. 如果目标目录已有历史快照或证据链，先读取作为上下文；不要用 RAG 代替本地文件检查。

### 2. 确认分析输入是否足够

开始分析前，必须确认或记录以下信息：

- Bug 现象。
- 用户观察到的准确时间点；如果没有时间点，则从全量日志中按现象反查。
- 时间误差范围；未提供时默认按 `±1 分钟` 搜索，无法确定时间点时扩大到全量日志。
- 目标日志目录。
- 项目目录。
- 模块；未提供时默认“云音乐模块”或“喜马拉雅”模块。

如果 Jira、当前工作目录和日志内容不足以支撑分析流程，则先向用户寻求帮助，例如询问具体现象和对应详细时间点，再继续。

### 3. 检查历史分析

1. 在目标目录查找：
   - `*_context_snapshot.md`
   - `*_evidence_chain.md`
   - `snapshot*.md`
   - 用户指定的历史分析文件
2. 如果存在，必须先读取并总结：
   - 之前的根因判断。
   - 关键日志。
   - 已修改代码。
   - 未确认项。
3. 新分析要明确说明本次与历史分析的相同点、差异点和是否推翻旧结论。

### 4. 日志分析

1. 优先围绕用户时间点按 `±1 分钟` 搜索关键 tag、关键词和业务对象。
2. 如果用户没有提供时间点，必须从全量日志中搜索符合现象描述的事件，并标明时间点由日志反查。
3. 还原事件时间线，至少包含：
   - 用户动作。
   - 关键请求/回调。
   - 状态写入。
   - 状态被覆盖。
   - 异常出现时的当前状态。
4. 对关键状态做反向追踪：
   - 谁第一次写入这个状态？
   - 谁最后一次覆盖它？
   - 本该出现的清理/LOSS/失败/取消/重置日志是否出现？
   - 如果没有出现，继续往更早时间段找。
5. 对并发、竞态、缓存类问题，必须查两个方向：
   - 成功样本中该状态如何正常转换。
   - 失败样本中哪一步没有发生。
6. 如果发现关键事件发生在当前时间窗口之前，必须主动扩大时间范围；不要只分析用户给出的表面异常时间点。

### 5. 源码分析

1. 从日志 tag、类名、方法名反查源码。
2. 优先看和日志直接对应的文件，不做无目的全项目搜索。
3. 每个关键日志必须尽量对应到：
   - 类/方法。
   - 条件分支。
   - 读写的变量或缓存。
   - 对外发送的事件或状态。
4. 如果日志显示状态被信任为真，必须找到代码里“为什么信任它”的判断。
5. 如果代码里有缓存命中、同曲 return、提前 return、焦点本地集合、全局单槽状态等，必须作为重点风险点继续追问来源。

### 6. NG 分析规则

如果用户说明这是 NG 分析：

1. 先分类：
   - A. 原现象完全没变。
   - B. 原现象改善但仍有残留。
   - C. 原现象消失，出现新问题。
2. 必须先验证测试包是否包含修复 commit；无法验证时，把“测试包是否包含 commit”列为最高优先级待确认项。
3. 必须读取原 Bug 快照，并对比：
   - 原关键日志是否仍出现。
   - 原根因链路是否仍成立。
   - 修复代码对应日志是否出现。
   - 新失败点发生在原失败点之前还是之后。

### 7. 保存分析快照与变更产物

分析完成后加载 `snapshot` skill，并在目标目录保存：

1. `<票号>_context_snapshot.md`
2. `<票号>_evidence_chain.md`
3. `<票号>_patch.diff`
4. `<票号>_commit_msg.md`

其中：

- `patch.diff` 记录当前项目最新代码相对于本次处理结果的补丁。
- `commit_msg.md` 记录根据项目 `AGENTS.md` 规则、结合本次 bug 写出的提交信息。

快照必须包含：

- 问题概述。
- 复现条件。
- 时间线。
- 根因分层。
- 关键日志证据。
- 源码链路。
- 修复思路或已实施修复。
- 待确认项。

保存到 `D:\logAnalyseForRag` 或其子目录后，执行知识库更新脚本：

```text
cmd.exe /c "D:\aboutMCP\rag-mcp-server\update_knowledge_base.bat"
```

并把脚本结果告知用户。

## 分析支撑规则

### 日志自动展开与筛选

单票分析必须尝试自动展开和筛选日志，不得只停留在附件下载完成。

1. 先枚举票据目录下的附件和压缩包，识别 `.zip`、`.7z`、`.rar`、`.tar`、`.gz`、分卷压缩包、Android logcat、aplog、crash、tombstone、ANR、dumpsys、window、package 等文件。
2. 对压缩包优先读取目录清单，按需抽取关键日志；避免无目的全量解压超大包。遇到分卷包时从第一卷读取目录并抽取需要的成员。
3. Android 日志优先关注 `aplog` / `logcat` / `main` / `system` / `events` / `radio` / `crash` / `dumpsys` / `window focus` / `package`；gzip 日志必须直接搜索或先解压到工作目录。
4. 自动生成可复用的中间筛选文件，例如 `work/<模块>_hits.txt`、`work/connectivity_hits.txt`、`work/<关键tag>_samples.txt`，但这些文件只能作为索引，不能替代原始日志证据。
5. 第一轮关键词从 Jira 现象和模块推导：应用包名、Activity/Fragment、业务 tag、错误文案、接口域名、状态枚举、异常类、网络/焦点/播放/登录/缓存关键词。
6. 如果第一轮命中太少，必须扩大关键词：包名前缀、模块通用 tag、系统服务 tag、UI 文案、接口路径、状态码、用户动作词；如果命中过多，必须按时间、进程号、tag、业务对象二次收敛。
7. 日志编码异常时仍要保留时间、进程、tag 和关键字段；中文乱码不能作为放弃分析的理由。
8. 结论引用必须回到原始日志行，包含时间、文件名或日志分片、进程/线程、tag、核心字段。

### 源码映射经验

日志分析和源码分析必须交替推进，不允许只看一边。

1. 从日志 tag、类名、方法名、状态枚举、接口 path、UI 文案反查源码。
2. 找到源码后必须反向解释日志：哪个方法打印、哪个分支进入、哪个变量导致、后续状态如何通知 UI 或服务。
3. 对“有网但提示无网 / 加载异常”类问题，必须同时查三条链路：业务接口是否真实成功、应用网络判断是否前置拦截、系统网络 validation/captive portal 是否异常。
4. 如果业务接口成功但 UI 失败，重点追 UI 状态机、LiveData/Flow、缓存状态、成功标志聚合、提前 return、重试逻辑。
5. 如果系统状态与业务可达性冲突，必须区分“系统 validation 未通过”和“业务域名不可达”，不要直接写成无网。
6. 每个根因结论至少要有一条日志证据和一段源码逻辑互相支撑；只有日志或只有源码时，置信度最高只能为中。

### 总原则

1. 只基于真实日志、代码和用户给出的现象下结论；没有证据的判断只能写入“待确认项”。
2. 分析时必须区分四层：
   - 现象：用户看到什么异常。
   - 直接原因：哪个状态/回调/分支导致异常出现。
   - 上游原因：为什么这个状态会被写错、没清掉、没回调、没互斥。
   - 最早根因：从当前日志和代码能证明的最早异常点。
3. 禁止把“直接原因”包装成“根因”。如果只能解释后半段，必须明确说“目前只解释到直接原因，还需要继续往前追”。
4. 对任何缓存、状态位、全局变量、焦点状态、播放态、登录态、网络态，都必须反查：
   - 最后一次被置为正确值/错误值的日志。
   - 应该被清掉/覆盖/回调但没有发生的日志。
   - 后续代码为什么相信这个缓存。
5. 如果继续分析需要更多日志、源码、时间范围、复现步骤、commit 信息或测试包信息，必须立即告诉用户，不要硬凑结论。

### 强制追根因规则

每次自动分析必须进行“5 Why”式追问，直到满足以下任一停止条件：

1. 找到最早可证明异常点，并能用日志或源码解释它如何导致用户现象。
2. 已向前追到日志起点，仍缺少关键事件，必须列出缺少的时间段和关键词。
3. 相关模块源码不在当前工作区，必须列出缺少的文件/模块/包名。
4. 系统或外部服务行为无法从应用日志证明，必须明确标为“需要系统/平台侧确认”。
5. 用户明确要求停止继续追因。

输出结论前必须自检以下问题：

- 我解释的是用户现象，还是最早根因？
- 当前结论是否还能继续问“为什么”？
- 如果还能继续问，是否已经查过对应日志和源码？
- 是否存在一个状态被错误缓存、未清理、未互斥、未回调？
- 是否需要更早日志或其他模块日志？如果需要，是否已经明确告诉用户？

## 输出要求

1. 先给结论，再给证据。
2. 结论必须标注置信度：高 / 中 / 低。
3. 关键日志引用要足够精确，包含时间、tag、核心字段。
4. 不要为了快速回答省略“不确定项”。
5. 如果还需要继续追根因，直接说“当前只定位到直接原因，我需要继续查更早日志/某模块源码”，不要假装已经闭环。
