# Log To Call Chain

当用户给你一份 Android App 日志，并希望你根据日志理清“用户做了哪些操作，以及这些操作让应用走了哪些方法链”时，使用这个 skill。

这个 skill 的目标很明确：

1. 先还原用户操作顺序
2. 再从日志中抽出每一步操作触发的关键事件
3. 最后把每一步操作对应的方法链梳理出来

默认输出重点是：**用户操作 + 对应方法链**。

不默认扩展到：

- bug 根因分析
- 修复方案设计
- 性能优化建议
- 架构重构建议

除非用户明确要求，否则只聚焦“操作如何走到哪些方法”。

## 适用场景

在以下情况优先使用：

- 用户贴了一段 Android 日志，希望看懂点击链路
- 用户想知道“我点了这个按钮后代码走了哪些方法”
- 用户想把一次页面操作映射成方法调用链
- 用户只需要把方法链梳理出来，不要求改代码
- 用户给的是操作时序日志、埋点日志、业务日志、页面生命周期日志、路由日志或混合日志

## 核心任务

你需要完成三件事：

1. **识别用户操作**
   从日志时间线中识别用户做了什么，例如进入页面、点击按钮、切换 tab、返回上一页、触发弹窗、提交表单。

2. **定位关键代码节点**
   找出与这次操作直接相关的类、方法、生命周期、监听器、路由跳转、ViewModel、Presenter、UseCase、Repository、网络请求、回调或状态分发节点。

3. **梳理方法链**
   用时间顺序和调用顺序，把一次操作对应的方法链写出来。

## 解释对象

默认把用户当成需要快速理解链路的 Android 开发者。

回答重点不是“日志里看到了什么”，而是：

- 用户做了哪一步操作
- 这一步操作触发了哪几个关键方法
- 方法之间是如何串起来的
- 哪些方法是入口，哪些方法是中间转发，哪些方法是真正执行点

## 工作原则

### 1. 先按时间线，不要一上来按模块讲

先把用户操作顺序理清，再把每步操作挂到方法链上。

优先回答：

- 用户先做了什么
- 随后日志出现了什么关键事件
- 这些事件对应哪条方法链

### 2. 区分“用户操作”和“系统噪音”

日志里可能混有大量无关内容，例如：

- GC
- 渲染抖动
- 三方 SDK 心跳
- 网络线程通用日志
- 系统广播
- 输入法日志

这些内容默认不是主链路，除非它直接参与当前操作的方法链。

### 3. 方法链必须有证据基础

每条链路都要尽量基于以下一种或多种证据：

- 日志里直接打印了方法名
- 日志里出现了类名、tag、页面名、Fragment、Activity、ViewModel、Presenter 名称
- 日志里出现了事件名、埋点名、路由路径、按钮标识、请求标识
- 代码中可以顺着这些线索找到调用路径

如果某一段是推断出来的，必须明确标注：

- “日志直接可见”
- “根据命名和调用位置推断”
- “需要代码进一步确认”

### 4. 以“操作”为单位输出，而不是只罗列方法

错误做法：

- `onClick()`
- `loadData()`
- `request()`

正确做法：

- 用户操作：点击“播放”按钮
- 对应方法链：`PlayerFragment.onClick()` -> `PlayerViewModel.play()` -> `AudioRepository.startPlay()` -> `XmPlayerManager.play()`

## 处理步骤

拿到日志后，按这个顺序工作。

### 第一步：识别日志边界和上下文

先确认：

- 这份日志覆盖的是哪一段操作时间
- 起点页面或起点状态是什么
- 终点页面或终点结果是什么
- 用户是否已经描述了操作步骤

如果用户已经说了“我先点 A，再点 B，再返回”，优先用用户描述作为主时间线，再用日志做证据补全。

### 第二步：按时间顺序抽取关键操作点

在日志中找出能代表操作切换的关键点，例如：

- `Activity` / `Fragment` 生命周期变化
- 点击事件日志
- 埋点事件
- 路由跳转
- 弹窗显示/关闭
- 请求发起与回调
- 状态切换日志
- RecyclerView item 点击
- Compose/传统 View 的事件回调

把这些关键点串成一条粗粒度操作时间线。

### 第三步：为每个操作找入口方法

每个用户操作都要先找“入口”。常见入口包括：

- `onClick()`
- `setOnClickListener` 绑定的方法
- `onItemClick()`
- `onMenuItemSelected()`
- `onCheckedChanged()`
- `onTabSelected()`
- `onResume()` 触发的初始化逻辑
- `navigate()` / `startActivity()` / `show()`
- MVI/MVVM 中的 `dispatchIntent()`、`processIntent()`、`handleAction()`

如果日志没有直接打印入口方法，就从页面、控件 ID、事件名、路由名去代码中找最可能的入口。

### 第四步：顺着入口往下梳理方法链

找到入口后，继续往下梳理：

- 谁调用了业务层
- 谁负责状态转换
- 谁负责请求数据
- 谁负责页面跳转或 UI 更新
- 谁是最终执行点

常见链路形态包括：

- `View -> ClickListener -> Fragment/Activity -> ViewModel -> Repository`
- `Fragment -> Presenter -> Interactor -> Repository -> Api`
- `Compose UI -> Event callback -> ViewModel -> UseCase -> Repository`
- `Router -> Activity/Fragment -> init/load -> request/render`

### 第五步：按“用户操作 -> 方法链”输出

最终必须把结果整理成用户能直接看的结构，而不是分析草稿。

## 输出要求

默认按下面结构输出。

### 1. 操作总览

先用 3 到 8 条，按时间顺序列出用户做了哪些主要操作。

示例：

```md
操作总览：
1. 进入首页
2. 点击“推荐”卡片
3. 进入详情页
4. 点击“立即播放”
5. 返回上一页
```

### 2. 每一步操作对应的方法链

每一步都按下面格式写：

```md
1. 用户操作：点击“立即播放”

- 日志证据：
  - `PlayerFragment: play button clicked`
  - `PlayerViewModel: dispatch PlayIntent`
  - `AudioRepository: startPlay trackId=123`

- 方法链：
  `PlayerFragment.onClick()`
  -> `PlayerViewModel.dispatchIntent()`
  -> `PlayerViewModel.handlePlay()`
  -> `AudioRepository.startPlay()`
  -> `XmPlayerManager.play()`

- 说明：
  入口在 `PlayerFragment` 的点击事件。
  真正开始播放的执行点是 `XmPlayerManager.play()`。
```

### 3. 如果某一步只能部分还原，要明确标注

例如：

- “这一步的入口日志缺失，但从后续 `DetailFragment` 初始化日志看，前面应当经过了 `Router` 跳转。”
- “日志能确认进入了 `ViewModel.loadData()`，但 `Repository` 之后的具体方法名需要进一步读代码确认。”

## 输出风格约束

- 先给操作顺序，再给方法链
- 方法链尽量短而关键，不要把无关辅助方法全塞进去
- 优先保留真正影响流程的节点
- 对 Android 常见组件要说清角色：`Activity`、`Fragment`、`ViewModel`、`Presenter`、`Repository`、`Router`
- 如果链路跨线程，可以注明主线程切到工作线程的位置
- 如果链路跨页面，要明确指出跳转节点

## 证据等级

输出时，尽量区分证据强弱：

- **直接证据**：日志里直接有类名、方法名、事件名
- **高可信推断**：日志里有页面名、tag、埋点、路由，代码里能顺着定位
- **待确认推断**：只能大致判断模块，具体方法名还需要进一步确认

如果是推断，不要装作已经完全确定。

## 禁止事项

- 不要只复述日志原文
- 不要只按模块解释，不按用户操作顺序解释
- 不要把系统噪音当成主链路
- 不要把所有打印过的方法都塞进链路
- 不要在用户没要求时扩展成 bug 修复方案
- 不要虚构不存在的调用关系

## 回答模板

```md
操作总览：
1. ...
2. ...
3. ...

1. 用户操作：...
- 日志证据：
  - ...
  - ...
- 对应方法链：
  `A()` -> `B()` -> `C()` -> `D()`
- 说明：
  入口是 ...，真正执行点是 ...。

2. 用户操作：...
- 日志证据：
  - ...
- 对应方法链：
  `E()` -> `F()` -> `G()`
- 说明：
  这里 ...。

补充说明：
- 哪些链路是日志直接确认的
- 哪些链路是根据代码结构推断的
```

## 成功标准

这个 skill 生效时，回答应满足以下标准：

1. 用户能一眼看出自己的操作顺序
2. 每个关键操作后面都能接上对应方法链
3. 输出的是“用户操作 -> 方法链”的映射，而不是散乱日志摘要
4. 结论能区分哪些是日志直接证据，哪些是代码推断
